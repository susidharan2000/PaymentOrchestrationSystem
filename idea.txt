Payment Processing System

*************************************************** Version 1 ************************************************************

A payment system is a machine that records irreversible money facts exactly once, despite retries, crashes, lies, and reordering.

******This system guarantees correctness per payment intent.**********

About the System

1)A payment system is a distributed correctness engine whose only hard rule is:
 * Never lose money. 
 * Never create money. 
 * Never apply effects twice.(no duplicate)
 * exactly-once recording of facts
 * at-most-once money movement
  even every thing crach


2)System prioritize the side effect (money movement) happens atmost once

3)A payment system is a (distributed state machine + ledger) that survives retries, crashes, timeouts, and lies from the network.

4)A payment is a controlled transfer of value between two ledgers.
5)Payment - coordinated ledger updates with irreversible consequences

6)Payments are:
   independent
   irreversible
   correctness-critical

7)Ledger = list of irreversible money facts

8) - Ledger entries must be created ONLY from:
     *Webhook events
     *Reconciliation results
   - NOT from:
     *API call return values
     *Worker assumptions
     *Timeouts

8)“stuck forever” as acceptable   **I will resolve stuck payments using a timeout policy, not by ledger facts.**
  liveness ≠ correctness
  waiting is safer than guessing
  timeouts are policy, not truth

  Retries are liveness optimizations not correctness

  Retries, timeouts, and UX fallbacks are layered on top of a correctness-first core.

  Add the retry policy last (Retries, timeouts, and UX fallbacks are layered on top of a correctness-first core.)

Correctness is presrved
Reconciler may leave state stale, but it must never lose or duplicate a ledger fact.


(Problem 1)How the unresolved payment are claimed and avoid  duplicate claim ?
Solution : leasing pattern
      In a background reconciler that repeatedly processes unresolved payments using LIMIT

      1)Concurrency control (avoid duplicate processing)
         * Use a lease field like claimed_at.
         * Eligibility condition:
         * claimed_at IS NULL OR claimed_at < now() - lease_timeout
         * This prevents multiple reconciler instances from processing the same row concurrently and ensures crash recovery via lease expiry.

      2)Fairness (avoid starvation)
         * Use deterministic ordering:
         * ORDER BY updated_at ASC
         * Without ORDER BY, repeated LIMIT queries on a persistent backlog can repeatedly return the same rows due to physical scan order, causing other rows to never get processed.

      * claimed_at → infrastructure coordination
      * updated_at → business freshness & fairness ordering

      * Lease controls visibility.
      * Ordering controls rotation.
      * Ledger idempotency guarantees correctness.

REFUND

 * System supports:
    * Full Refund
    * Partial Refund
    * Multiple Partial Refund

 * Duplicate refunds are prevented using an idempotency_key

 * The external system that requests the refund manages the idempotency_key

 * A refund is allowed only when:
      refund_amount <= remaining_refundable

 * remaining_refundable =
      captured_amount - refunded_amount

 * The refund is processed asynchronously because the user will not wait on the page for the refund

 * Refund completion may be notified via:
      * SMS
      * Email
      * Webhooks
      * Merchant dashboard updates

Data Base Schema

1)payment_intent

    * payment_id - Primary Key  NOT NULL
    * idempotency_key  NOT NULL

    * state - CHECK (
        CREATED,
        PROCESSING,
        CAPTURED,
        FAILED,
        CANCELLED,
        PARTIALLY_REFUNDED,
        REFUNDED
      ) NOT NULL

    * amount NOT NULL (original intent of the payment)
    * captured_amount NOT NULL DEFAULT 0   (how much money actually moved from the customer to the merchant.)
    * refunded_amount NOT NULL DEFAULT 0 (how much money has been returned to the customer.)

    * currency NOT NULL

    * psp_ref_id NULL
    * psp_name NOT NULL

    * request_hash NOT NULL

    * created_at NOT NULL

    * reconciler_claimed_at NULL
    * reconcile_attempts DEFAULT 0
    * last_reconcile_error NULL

    UNIQUE(idempotency_key)
    UNIQUE(psp_ref_id)

2)ledger_entries (Append only) - frobid DELETE and UPDATE with triggers  
   * ledger_entry_id NOT NULL  Primary Key 
   * entry_type - CAPTURED / FAILED
   * payment_id  REFERENCES  payment_intent(payment_id)
   * amount 
   * currency
   * psp_ref_id 
   * psp_name 
   * created_at
   UNIQUE (psp_name, psp_ref_id) // to support correctness accross multiple PSP's

3) refund_record
   * refund_entry_id NOT NULL
   * idempotency_key  NOT NULL
   * status - PENDING / PROCESSING / SUCCEEDED / FAILED
   * payment_id  REFERENCES  payment_intent(payment_id)
   * amount
   * currency
   * psp_payment_ref_id
   * psp_refund_id
   * created_at
   UNIQUE(idempotency_key)
   UNIQUE (psp_refund_id)



Retry Policy 
* The system decides “stuck payments” using time, not truth.
* A timeout may mark a payment FAILED as a policy decision, not a truth assertion.
* sA user retry creates a new payment intent; multiple successes are acceptable and resolved later via refunds, not by blocking truth.

Principle:
The system is designed so mistakes are detectable and fixable, not hidden.
